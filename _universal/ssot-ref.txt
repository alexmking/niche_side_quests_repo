=== CONVENTIONS ===
================================================================================

  This file is the single source of truth for notes, commands, how-tos, and
  definitions. The rules below keep it consistent and grep-friendly.

  SECTION HEADERS
    Top-level section  →  === SECTION NAME ===         (one blank line above/below)
    Subsection         →  --- Subsection Name ---      (one blank line above/below)

  ENTRY TYPES  (prefix every standalone entry with one label)
    [CMD]   Shell command or CLI invocation
    [HOW]   Step-by-step instructions / procedure
    [DEF]   Definition or explanation of a term or concept
    [NOTE]  Personal note, observation, reminder, or opinion

  INLINE TAGS  (append @tagname anywhere on a line to enable cross-section grep)
    Example:  [NOTE] Prefer rebasing over merging for feature branches  @git @workflow

  CODE / COMMANDS
    Indent with 2 spaces.
    Prefix shell commands with $, output lines with >.
    Example:
      $ git log --oneline -10
      > abc1234 fix: typo in readme

  DATE STAMPS  (for time-sensitive or versioned entries)
    Format: [YYYY-MM-DD]
    Example:  [CMD] [2026-06-12] Upgrade Homebrew after macOS 15 migration  @brew

  SEPARATORS
    Blank line between individual entries.
    --- subsection header --- between groups within a section.
    === section header === between top-level topics.

  GREP CHEATSHEET
    All section headers:    grep "^=== " ssot-ref.txt
    All subsections:        grep "^--- " ssot-ref.txt
    All commands:           grep "\[CMD\]" ssot-ref.txt
    All how-tos:            grep "\[HOW\]" ssot-ref.txt
    All definitions:        grep "\[DEF\]" ssot-ref.txt
    All notes:              grep "\[NOTE\]" ssot-ref.txt
    By tag:                 grep "@git" ssot-ref.txt
    Entries with a date:    grep "\[20[0-9][0-9]-" ssot-ref.txt

   ### SNIPPETS prefix approach:
   - semicolon ; is the designated leading-char for most/all SNIPPETs, like so:  ;zzshorts  ||  ;keybvis
   - the semicolon is *ONLY* for SNIPPETS. So you cant use it with any quicklink or raycast-alias defintions.
      - this way, we don't have to worry about triggering snippets on accident while in middle of raycast-search while at the same time still being able to use very short snippet triggers like 'pzz'
      - since 'pzz' can be the raycast alias, and ';pzz' can be the snippet
   - HOWEVER, the above prefixes like 'p' or 'path' or 'edit' etc CAN be used for quicklink/aliases.
   - We should have multiple snippets for the same shortcut/action so it's easier to trigger on the fly w/o needing to remember the exact single one (as long as the mult snippets dont interfere with the quickLink/alias equiv)
      - the STANDARD snippet is the one with the leading ';', such as   ;zzshort    .... or ... ';pzpz' 
         - Note because the context of SNIPPETS are already known, you dont really need the prefix like quicklink-names and aliases kinda do (like typing zzshorts in rayc-search, it's not clear what you want to do with zzshorts, which is why a prefix like 'p' or 'edit' is needed)
            - However, with Snippets, you alraedy know you're trying to paste some string, so there's no need to define the snippet like ;pathzzshorts or ;pzshorts since the filename alone is enough to remove any ambiguity
